# 01 - 系统概览

## 1. 产品定义

本项目不是“几个 LLM 假装在玩阿瓦隆”。

本项目是一个 **后端优先的阿瓦隆多 Agent 游戏平台**，需要同时承担以下职责：

1. 执行一套**正式定义、可版本化**的阿瓦隆规则集
2. 在严格的可见性边界下编排真人玩家与/或 LLM 玩家
3. 持久化完整的游戏事件、审计信息、回放数据与恢复数据
4. 为未来扩展新角色、新规则、新人数、新 UI、新联网能力预留干净接口

---

## 2. 产品阶段规划

### V1：纯 LLM 自动对局
- 不需要前端
- 可创建一局游戏并自动跑到结束
- 所有玩家都由 LLM 控制
- 每轮都要能查看：
  - 当前公开局面
  - 每个玩家的公开发言
  - 每个玩家的结构化动作
  - 每个玩家的审计摘要

### V2：单机人机模式
- 用一个真人玩家替代一个 LLM
- 轮到真人时系统进入等待态
- 其余流程仍由系统自动推进

### V3：多人联机模式
- 多个真人玩家可加入
- 缺失席位可由 LLM 补位
- 使用 WebSocket 或 SSE 推送实时事件

---

## 3. 不可妥协的设计约束

### 3.1 规则权威
合法性判断、视野判断、阶段推进、胜负判定，全部由规则引擎决定。  
**绝不能信任 LLM 自己执行规则。**

### 3.2 玩家视野隔离
每个玩家只能看到：
- 当前规则允许看到的信息
- 当前身份允许知道的信息
- 当前阶段允许知道的信息

### 3.3 全量可审计
每一步关键行为都必须存为结构化事件和审计记录。

### 3.4 可恢复
进程中断后，必须能依赖：
- 事件流
- 游戏快照
- 玩家记忆快照  
恢复继续运行。

### 3.5 可扩展
系统必须能扩展：
- 更多人数
- 更多角色
- 更多规则变体
- 更多玩家控制器类型
- 更多 LLM provider / model

---

## 4. 核心概念

### 规则集（Ruleset）
一套带版本号的“本局游戏怎么玩”的正式定义。

### 角色目录（Role Catalog）
角色的元数据描述，包括：
- 阵营
- 可见性
- 被动特性
- 行动能力
- 私有知识规则

### 开局模板（Setup Template）
从“玩家人数”映射到“角色组合”的模板。

### 游戏会话（Game Session）
一局具体可运行的游戏实例。

### 玩家控制器（Player Controller）
一个抽象控制器，具体可以是：
- LLM 驱动
- 真人驱动
- 脚本驱动

### 玩家记忆（Player Memory）
每个玩家独立私有状态，用于保存：
- 信念
- 怀疑分
- 承诺
- 观察结论
- 当前策略模式

### 审计事件（Audit Event）
对编排器、玩家或规则引擎某次动作的持久化记录。

---

## 5. V1 不应该优先做的事

不要在第一版过度建设这些内容：

- 可视化后台管理 UI
- 在线规则录入页面
- 跨局长期记忆系统
- 重型 RAG 流水线
- 语音聊天
- 匹配大厅
- 社交系统

V1 应优先保证：

- 规则正确
- 状态可追踪
- 行为可审计
- 尽量确定性
- 领域模型可扩展
- 控制器抽象清晰

---

## 6. 给开发 Agent 的说明

### 开发目标
先做一个 **可严肃运行和审计的后端引擎**，而不是一个“能说几句台词”的 AI demo。

### 开发优先级
1. 规则定义与规则引擎
2. 角色与视野系统
3. 开局分配与状态机
4. 事件流、快照、恢复
5. 控制器抽象
6. LLM 接入

### 明确禁止
- 禁止让 Prompt 成为规则真相源
- 禁止让 LLM 决定阶段推进
- 禁止把全部状态只放在模型上下文中
- 禁止先写 UI 再补后端真相层
