# 06 - 持久化、审计、回放与恢复

## 1. 为什么这层是核心，不是附属

你想要的不是“AI 们玩一局然后结束”。  
你要的是：

- 每步都可追踪
- 赛后可复盘
- 中断后可继续
- 未来可接前端
- 未来可做性能评测和模型对比

这些都依赖持久化、审计、回放与恢复层。

---

## 2. 事件流（Event Store）

### 设计原则
所有改变游戏真相的行为都必须写事件。

### 推荐事件类型
- `GAME_CREATED`
- `PLAYERS_REGISTERED`
- `ROLE_ASSIGNMENT_COMPLETED`
- `ROLE_REVEALED_TO_PLAYER`
- `ROUND_STARTED`
- `DISCUSSION_STARTED`
- `PLAYER_PROMPT_PREPARED`
- `PLAYER_PUBLIC_SPOKE`
- `PLAYER_ACTION_SUBMITTED`
- `TEAM_PROPOSED`
- `TEAM_VOTE_CAST`
- `TEAM_VOTE_RESOLVED`
- `MISSION_ACTION_CAST`
- `MISSION_RESOLVED`
- `ASSASSINATION_STARTED`
- `ASSASSINATION_SUBMITTED`
- `GAME_ENDED`

### 推荐事件结构

```java
public class GameEvent {
    private String eventId;
    private String gameId;
    private Long seqNo;
    private String type;
    private String phase;
    private String actorPlayerId;
    private String payloadJson;
    private Instant createdAt;
}
```

---

## 3. 审计记录（Audit Record）

事件解决的是“发生了什么”。  
审计记录解决的是“为什么这样做”“调用时上下文是什么”。

### 推荐审计字段

```java
public class AuditRecord {
    private String auditId;
    private String gameId;
    private Long eventSeqNo;
    private String playerId;
    private String visibility; // PUBLIC / PRIVATE / SYSTEM / ADMIN_ONLY
    private String inputContextJson;
    private String inputContextHash;
    private String rawModelResponseJson;
    private String parsedActionJson;
    private String auditReasonJson;
    private String validationResultJson;
    private String errorMessage;
    private Instant createdAt;
}
```

---

## 4. 游戏快照（Game Snapshot）

如果每次恢复都全量重放全部事件，随着局数变大和轮次增多会越来越慢。

因此建议定期保存当前局面快照。

```java
public class GameSnapshot {
    private String gameId;
    private Long basedOnEventSeqNo;
    private Integer roundNo;
    private String phase;
    private String publicStateJson;
    private Instant createdAt;
}
```

### 建议策略
- 每 N 个事件保存一次快照
- 每一轮结束后至少保存一次
- 关键阶段切换时优先保存

---

## 5. 玩家记忆快照（Player Memory Snapshot）

玩家记忆是私有状态，必须单独持久化，不能只依赖下一次现算。

```java
public class PlayerMemorySnapshot {
    private String snapshotId;
    private String gameId;
    private String playerId;
    private Long basedOnEventSeqNo;
    private String memoryJson;
    private Instant createdAt;
}
```

### 这样做的价值
- 中断恢复时可直接读取
- 不依赖模型“记得之前说过什么”
- 为未来人机混合、断线重连、长局扩展打基础

---

## 6. 回放投影（Replay Projection）

未来 UI 不应直接扫原始事件流做渲染，最好构建一个可查询的回放投影。

例如：
- 某一轮每个人说了什么
- 某一步队伍怎么被组出来
- 某轮投票分布
- 哪轮任务失败
- 刺客为何锁定某人

### 推荐做法
建立只读 projection 表或缓存视图。

---

## 7. 恢复服务（Recovery Service）

### 恢复流程
1. 读取最新 `GameSnapshot`
2. 读取 snapshot 之后的事件
3. 重建当前运行态
4. 读取所有玩家最近的 `PlayerMemorySnapshot`
5. 将系统恢复到可继续 step 的状态

### 注意
恢复后系统必须知道：
- 当前 phase
- 当前轮到谁
- 当前允许动作是什么
- 哪些玩家在等待真人输入
- 每个 LLM 玩家的当前记忆是什么

---

## 8. 数据库层建议

### 必备表
- `game_session`
- `game_player`
- `role_assignment`
- `game_event`
- `audit_record`
- `game_snapshot`
- `player_memory_snapshot`
- `replay_projection`

### 建议
- 用 Flyway 或 Liquibase 管理迁移
- 重要 JSON 字段保留 schema version
- `game_id + seq_no` 建唯一索引
- 审计与事件最好可按 game_id 顺序查询

---

## 9. 给开发 Agent 的实施要求

### 必做
- 先落事件表，再落快照表
- 事件写入必须幂等或至少可检测重复
- 所有恢复逻辑必须有集成测试

### 推荐顺序
1. `game_event`
2. `game_snapshot`
3. `player_memory_snapshot`
4. `audit_record`
5. `RecoveryService`
6. `ReplayQueryService`

### 禁止
- 禁止只把状态放内存
- 禁止只存最终结果，不存中间步骤
- 禁止恢复逻辑依赖 LLM 重新“回忆”
