# Compound Learnings 2026-03-30 Seed Randomization

## ce-brainstorm

- 当前问题不是“随机算法失效”，而是“默认入口把随机种子固定了”，所以所有基于 seed 的洗牌结果都会稳定复现。
- Avalon 这里至少有三处初始随机面会受同一 seed 影响：角色分配、随机模型池分配、首位队长选择。只修其中一处，仍会留下“每局开面很像”的错觉。
- `SCRIPTED` 控制器本身高度确定性，不能把“对局重复”全部归因到角色分配；但在默认 `role` 模式下，固定 seed 仍是最主要根因。

## ce-plan

- `seed` 应保留为公共接口字段，但语义要改成“显式传入才复现，省略则自动生成 effective seed”。
- 自动生成逻辑需要集中成共享抽象，而不是在 API 和控制台各写一套临时随机实现，否则后续测试和默认策略很难统一。
- 控制台如果允许“留空自动生成”，就必须把实际生效 seed 显示出来，否则复盘和问题重现会退化。

## ce-review

- 本次改动没有修改持久化 schema，也没有改事件结构；恢复链路仍通过 `setup.seed` 和 `GAME_STARTED.leaderSeat` 复原状态。
- 首位队长随机化被限制在“起局一次性确定”，没有扩散到后续轮转规则，因此不会额外放大回放和恢复风险。
- 针对 API 的无 seed 路径补了服务级和 Spring 集成级覆盖，确保不是只在控制台一侧看起来随机。

## ce-compound

- 当系统同时支持“可复现随机”和“默认随机”时，最稳妥的做法是把随机源前移成 `effectiveSeed`，之后所有派生随机行为都只依赖这个值。
- 控制台或 CLI 的“默认值”如果落在随机控制项上，实际上是在把产品默认行为从“随机”改成“固定场景”；这类默认值需要谨慎。
- 共享随机策略不要散落在 UI、应用服务和运行时各层；应先确定 seed，再让角色、模型和首位行动者都从同一个 seed 派生。
