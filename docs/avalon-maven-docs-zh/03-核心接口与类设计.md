# 03 - 核心接口与类设计

## 1. 目标

本文件给出项目第一批必须创建的核心接口与类。重点不是写完整实现，而是先把骨架定住，让 Codex / Claude Code 可以基于这些骨架继续补实现。

## 5. 核心接口

### `PlayerController`
统一玩家控制器接口。

建议方法：

- `PlayerActionResult act(PlayerTurnContext context)`

实现类：

- `LlmPlayerController`
- `HumanPlayerController`
- `ScriptedPlayerController`

### `GameRuleEngine`
规则引擎接口。

建议方法：

- `void validateAction(Game game, PlayerAction action)`
- `Game applyAction(Game game, PlayerAction action)`
- `Phase nextPhase(Game game)`
- `boolean isGameEnded(Game game)`

### `RoleAssignmentService`
建议方法：

- `List<RoleAssignment> assignRoles(Game game, RuleSetDefinition ruleSet, SetupTemplate template, long seed)`

### `TurnContextBuilder`
建议方法：

- `PlayerTurnContext build(Game game, Player player)`

### `PlayerControllerResolver`
建议方法：

- `PlayerController resolve(Player player)`

### `AgentGateway`
建议方法：

- `AgentTurnResult execute(AgentTurnRequest request)`

## 6. 运行时核心服务

### `GameOrchestrator`
职责：

- 决定当前应由谁行动
- 构造回合上下文
- 调用对应 controller
- 交由 rule engine 校验与应用动作
- 触发事件写入与快照保存

建议最少方法：

- `void step(String gameId)`
- `void runToEnd(String gameId)`

### `GameSessionService`
职责：

- 创建游戏
- 开始游戏
- 加载 / 保存当前会话状态
- 暂停、恢复

## 10. 给 AI 编码代理的要求

创建骨架时必须优先保证：

1. 所有关键类先有空实现或基础字段
2. 接口签名先稳定，不急于完善方法体
3. 所有 controller 与 engine 的依赖先通过接口解耦
4. 允许先返回占位实现，但类名、包结构、依赖方向必须正确
