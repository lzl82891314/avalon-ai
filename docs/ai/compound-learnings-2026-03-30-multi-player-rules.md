# Compound Learnings 2026-03-30 Multi Player Rules

## ce-brainstorm

- 这次需求不是“多加几份 YAML 配置”这么简单，而是要把整个规则系统从“默认只有 5 人局”提升为“标准阿瓦隆 5-10 人局均可运行”。
- 人数扩展会同时影响建局入口、角色池、任务轮次人数、特定轮次失败阈值、运行时规则查询、恢复逻辑和测试夹具；如果只改其中一层，系统就会出现“能建局但跑不完”或“能跑但恢复错误”的断层。
- 标准阿瓦隆多人局的关键不是任意组合角色，而是官方预设人数对应的固定角色组合；把它做成自由拼装会显著增加校验复杂度和回归面。
- `LOYAL_SERVANT` 这类重复角色会穿透配置校验、角色到模型的绑定校验以及控制台默认映射，重复角色支持必须是系统级能力，而不是单点例外。

## ce-plan

- 规则来源要统一到配置驱动模型，运行时和恢复逻辑不能继续依赖单独的 `ClassicFivePlayerGameRuleEngine` 之类硬编码实现。
- 控制台默认入口要按人数反推出 `ruleSetId` 和 `setupTemplateId`，这样操作路径才能与多人规则扩展保持一致；否则多人能力只存在于底层 API。
- 兼容策略应区分“默认入口升级”和“旧资源保留”两个层次：默认行为切到 `v2`，但显式引用旧 `v1` 资源的测试或调用方不应被无条件破坏。
- 规则合法性校验要补硬边界，例如忠臣不能执行 `FAIL`，否则人数扩展后虽然流程更完整，但核心阿瓦隆规则仍可能被 LLM 输出绕过。

## ce-review

- 本次改动没有修改持久化 schema，但运行时规则解析已经成为恢复链路的一部分，因此恢复逻辑必须和主运行时共享同一套规则引擎实现。
- 支持重复角色后，凡是基于“角色集合”的校验都必须先去重，再判断绑定覆盖范围；否则多人模板里合法的多个 `LOYAL_SERVANT` 会被误判为重复冲突。
- 控制台展示层不能再假定固定 5 个玩家或固定 5 轮静态文案，凡是依赖公共状态的玩家遍历和规则摘要都要从当前规则与 `playerCount` 动态生成。
- 多人规则扩展最容易遗漏的是测试资产和文档示例，它们虽然不影响主逻辑运行，但会快速把仓库重新拉回过期的 5 人默认假设。

## ce-compound

- 只要规则存在按人数变化的维度，就不要把人数逻辑散落在控制台、编排器、恢复器和测试夹具里；应由规则配置统一表达，再由单一规则引擎解释。
- “重复角色”不是配置文件层面的特殊情况，而是整个系统的数据建模要求；一旦支持标准阿瓦隆多人局，就必须让模板校验、角色绑定和运行时解析都接受重复角色。
- 对于带默认入口的系统，功能升级最容易失败在“底层已支持，但默认入口仍指向旧配置”；因此每次扩展规则范围时，都应显式检查 console、README、集成测试和样例请求是否同步升级。
- 当静态 model profile 文件名与 `modelId` 脱钩时，所有 API 与测试都应以 `modelId` 为准，而不是再把资源文件名当成外部稳定标识。
