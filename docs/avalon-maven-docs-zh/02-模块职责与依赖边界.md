# 02 - 模块职责与依赖边界

## 1. 目标

本文件定义每个模块该做什么、不该做什么，以及模块之间允许的依赖方向。这个边界必须在项目一开始就定死，否则后面很容易变成“AI 调用、规则判定、数据库写入、接口输出”全都缠在一起。

---

## 2. 依赖方向总图

推荐依赖方向：

```text
avalon-core <- avalon-runtime <- avalon-api <- avalon-app
      ^             ^
      |             |
      +------ avalon-agent
      +------ avalon-persistence
```

更准确地说：

- `avalon-runtime` 依赖 `avalon-core`
- `avalon-agent` 依赖 `avalon-core`，必要时可依赖 `avalon-runtime` 中的上下文模型
- `avalon-persistence` 依赖 `avalon-core`，必要时依赖 `avalon-runtime` 的快照模型
- `avalon-api` 依赖 `avalon-runtime`、`avalon-persistence` 与 `avalon-core`
- `avalon-app` 负责把所有实现装配到一起

禁止反向依赖。

---

## 3. `avalon-core` 职责边界

### 应包含

- `Game`
- `Player`
- `RoleDefinition`
- `RuleSetDefinition`
- `SetupTemplate`
- `RoleAssignment`
- `Phase`
- `GameStatus`
- `PlayerController`
- `GameRuleEngine`
- `PlayerTurnContext`
- `PlayerActionResult`
- `PlayerMemoryState` 的纯模型版本
- 规则相关值对象与异常

### 不应包含

- Spring 注解
- Repository
- Controller
- ChatModel
- Prompt 构造
- 数据库 Entity

### 核心原则

这里定义的是“系统真相的抽象”，而不是“实现方式”。

---

## 4. `avalon-runtime` 职责边界

### 应包含

- `GameOrchestrator`
- `TurnScheduler`
- `PlayerControllerResolver`
- `TurnContextBuilder`
- `GameSessionService`
- 阶段推进逻辑
- 控制器调用逻辑

### 不应包含

- 具体 LLM 调用细节
- 具体数据库 SQL / JPA 细节
- Web Controller

### 核心原则

runtime 是“调度器”，不是“裁判”也不是“模型调用器”。

- 规则判定交给 `GameRuleEngine`
- 模型调用交给 `PlayerController` 的具体实现
- 持久化交给 persistence 模块提供的 store / repository

---

## 5. `avalon-agent` 职责边界

### 应包含

- `LlmPlayerController`
- `AgentGateway`
- `SpringAiAgentGateway`
- `ModelProfile`
- `ChatModelFactory`
- Prompt 模板
- 输出 JSON 解析与校验
- 模型原始响应到 `PlayerActionResult` 的转换

### 不应包含

- 游戏胜负判定
- 回合推进主流程
- 事件流主表定义

### 核心原则

agent 模块只负责一件事：

> 给定一个玩家的回合上下文，调用该玩家绑定的模型，并返回结构化结果。

---

## 6. `avalon-persistence` 职责边界

### 应包含

- `GameEventEntity`
- `GameSnapshotEntity`
- `PlayerMemoryEntity`
- Repository
- Store 实现
- JSON 序列化持久化
- 快照恢复实现

### 不应包含

- REST API
- Prompt 组装
- 规则判定

### 核心原则

持久化层负责“保存”和“恢复”，不负责解释规则。

---

## 7. `avalon-api` 职责边界

### 应包含

- 创建游戏接口
- 开始游戏接口
- 单步推进接口
- 运行至结束接口
- 查询状态接口
- 查询事件接口
- 查询回放接口
- 提交人类玩家动作接口

### 不应包含

- 直接操作数据库 Entity
- 直接拼 Prompt
- 在 Controller 中写业务规则

### 核心原则

API 层只是外观层。

---

## 8. `avalon-app` 职责边界

### 应包含

- Spring Boot 启动入口
- Bean 装配
- 第三方框架配置
- 各模块 wiring

### 不应包含

- 业务规则代码
- 长逻辑实现

---

## 9. 横向抽象：PlayerController

这是整个系统最重要的可扩展点之一。

### 统一接口

- `LlmPlayerController`
- `HumanPlayerController`
- `ScriptedPlayerController`

三者都实现同一个接口。

这样才能支持：

- V1：全 LLM
- V2：1 人类 + 4 LLM
- V3：多人联机 + LLM 补位

AI 编码代理必须保证：

- 运行时只依赖接口，不依赖具体 controller 类型

---

## 10. 横向抽象：规则真相源

规则只能有一个真相源：

- `RuleSetDefinition`
- `RoleDefinition`
- `SetupTemplate`

任何 prompt 中的规则说明都必须由这些结构生成。

禁止做法：

- 手写一段 prompt 当成规则
- 每个 agent 使用不同规则文本

---

## 11. 横向抽象：事件与快照

系统恢复与审计依赖三类数据：

- 事件流
- 游戏快照
- 玩家记忆快照

这三个对象应由 persistence 模块管理，但模型定义可在 core / runtime 中抽象。

---

## 12. 给 AI 编码代理的边界校验清单

Codex / Claude Code 在生成代码时应自查：

1. `avalon-core` 中是否出现 Spring 注解
2. `avalon-api` 中是否直接操作数据库实体
3. `avalon-agent` 中是否包含游戏胜负判定
4. `avalon-runtime` 中是否直接 new 某个具体 LLM SDK client
5. 规则是否被写死在 prompt 而不是配置中

只要以上任一项为真，就说明设计已经开始偏离。
