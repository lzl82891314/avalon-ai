# 04 - 规则、角色与开局配置

## 1. 为什么规则必须是一等数据

不同阿瓦隆桌游版本规则并不完全相同。  
如果规则只写在 prompt 里，不同模型会在不知不觉中按不同版本理解游戏。

因此必须明确：

- 后端持有**唯一正式规则集**
- Prompt 中的规则说明必须从后端规则元数据生成
- 回放与审计必须记录具体使用了哪一个规则集版本

---

## 2. RuleSetDefinition（规则集定义）

```java
public class RuleSetDefinition {
    private String ruleSetId;
    private String name;
    private String version;
    private Integer minPlayers;
    private Integer maxPlayers;
    private List<String> supportedSetupTemplateIds;
    private List<TeamSizeRule> teamSizeRules;
    private VotingRule votingRule;
    private MissionRule missionRule;
    private AssassinationRule assassinationRule;
    private VisibilityPolicyDefinition visibilityPolicy;
    private SetupRule setupRule;
}
```

### 规则集至少要定义
- 支持多少玩家
- 支持哪些开局模板
- 每轮任务人数
- 某轮是否需要双失败
- 连续否决队伍上限
- 是否开启刺杀阶段
- 刺客与梅林角色标识
- 可见性是否按角色知识规则计算

---

## 3. RoleDefinition（角色定义）

不要写成巨型 `if (role == MERLIN)` 分支地狱。  
角色必须数据化、能力化。

```java
public class RoleDefinition {
    private String roleId;
    private String displayName;
    private Camp camp;
    private String description;
    private List<KnowledgeRuleDefinition> knowledgeRules;
    private List<ActionCapability> actionCapabilities;
    private List<PassiveTrait> passiveTraits;
    private Boolean canJoinMission;
    private Boolean canVote;
    private Boolean canLead;
    private Boolean canAssassinate;
}
```

### 角色定义负责说明
- 这个角色属于哪个阵营
- 开局能知道什么
- 具备什么行动能力
- 有哪些被动特性
- 是否能刺杀 / 组队 / 参与任务

---

## 4. SetupTemplate（开局模板）

开局模板负责把“人数”映射到“角色列表”。

```java
public class SetupTemplate {
    private String templateId;
    private Integer playerCount;
    private List<String> roleIds;
    private Boolean enabled;
}
```

例子：
- `classic-5p-v1`
- `classic-6p-v1`
- `classic-7p-v1`

### 设计价值
这样未来增加人数，不需要改规则引擎核心，只需要新增模板与少量规则配置。

---

## 5. 知识规则示例

知识规则描述“角色揭示阶段某个角色能看到什么”。

例如：
- Merlin 能看到除 Mordred 外的坏人
- Percival 能看到 Merlin 与 Morgana，但不知道谁是谁
- 坏人互相可见，但 Oberon 例外

这必须表示成**机器可读逻辑**，不是只靠自然语言说明。

## 6. 随机发身份

### 正确做法
由系统统一根据规则与模板生成角色牌组，再按 seed 洗牌分配。

### 错误做法
让每个 LLM 自己“抽”身份。

### 原因
系统必须保证：
- 不重复
- 不遗漏
- 可审计
- 可回放
- 可恢复

### 推荐服务

```java
public class RoleAssignmentService {
    public List<RoleAssignment> assignRoles(GameSetupContext context) {
        // 1. 加载模板
        // 2. 构造 role deck
        // 3. 根据 seed 洗牌
        // 4. 分配到每个 seat
    }
}
```

## 7. 推荐配置形式

V1 不需要图形化规则录入后台。  
推荐先使用：
- YAML
- JSON

系统启动时加载并做严格校验。

### 必须具备
- 配置文件校验失败即快速启动失败
- 角色引用必须能解析
- 模板角色数量必须等于人数
- 规则集引用的模板必须存在

---

## 8. 给 Prompt 的规则说明怎么来

不要手写一堆分散的 Prompt 文本。  
应该由系统根据 `RuleSetDefinition` 自动生成玩家可读规则摘要。

例如：

```text
本局规则集：Avalon Classic 5P v1.0.0
玩家人数：5
任务轮次人数要求：2, 3, 2, 3, 3
连续 5 次队伍投票被否决则坏人自动获胜
好人完成 3 次任务后进入刺杀阶段
刺客若刺中梅林，则坏人胜利
```

---

## 9. 给开发 Agent 的实施要求

### 第一优先级
先把以下三类配置模型做出来并跑通：
- `RuleSetDefinition`
- `RoleDefinition`
- `SetupTemplate`

### 任务完成标准
- 能加载一个 5 人经典局
- 能校验配置错误
- 能根据 seed 分配角色
- 能生成角色揭示信息

### 禁止
- 禁止把规则散落在 Prompt 模板中
- 禁止把角色能力写死在几十个 if/else 中
