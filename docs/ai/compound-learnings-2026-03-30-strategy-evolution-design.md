# Compound Learnings 2026-03-30 Strategy Evolution Design

## ce-brainstorm

- 当前问题不是“大模型不会思考”，而是“系统没有把思考结果变成可持续行为变化的闭环”。
- 阿瓦隆里的固定策略现象有两层根因：一层是 prompt 与角色视角的静态化，另一层是 runtime 缺少把 `memoryUpdate` 写回并继续利用的机制。
- `audit`、`decision report`、`player_memory_snapshot` 已经足够支撑一套轻量级策略进化系统；缺的不是数据，而是版本化、评估和晋升链路。
- 角色独立演化比全局共享经验池更稳妥，因为阿瓦隆不同角色的目标函数和风险结构明显不同。

## ce-plan

- 进化系统要分成两层：局内记忆闭环负责“这一局越玩越像在学习”，跨局策略版本化负责“下一局不再重复同一个坏习惯”。
- `PlayerAgentConfig.promptProfileId` 是现成的策略版本挂点，优先激活它比重新发明一套席位级字段更合适。
- 正式局与探索局必须分离：生产使用 champion，候选策略只在离线评估或灰度中探索，否则会把“实验噪声”直接引入正式结果。
- 策略优劣必须通过固定 seed 套件、座位轮换和角色指标比较来判断，不能凭单局观感或“看起来更随机”来下结论。

## ce-review

- 本次工作是设计文档落盘，没有改动 runtime、persistence、API 或 migration，因此不会引入运行时回归风险。
- 设计里把 `player_memory_snapshot` 保留为局内恢复设施，没有强行让它承担跨局版本管理，边界更清晰。
- 文档显式限制了 candidate 自动替换 champion，避免后续实现时把“自动生成候选”误扩展成“自动发布正式策略”。
- 把“派西维尔首轮双候选上车”写成专项评估案例，有助于后续实现者避免只做抽象平台而缺少可验证业务目标。

## ce-compound

- 当系统里已经存在 `memoryUpdate`、`audit`、`decision report` 这类观察产物时，最容易犯的错是把“有记录”误当成“会学习”；只有记录被正式写回并影响下一轮决策，学习闭环才成立。
- 对多角色博弈系统，策略进化的最小稳定单位通常不是“模型”而是“角色”；按角色维护 champion / candidate 能显著降低相互污染风险。
- 对 agent 系统而言，随机探索和策略进化必须分开治理：前者解决“看更多可能性”，后者解决“留下更好的行为”。
- 设计文档如果要交给后续实现者，应明确写出非目标和发布门禁；否则实现很容易从“策略平台”滑向“自动把实验版本推上生产”。
