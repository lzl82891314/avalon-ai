# Compound Learnings 2026-03-30 Seat Binding

## ce-brainstorm

- 这次问题不是“6 人局规则错了”，而是“显式模型绑定的主语选错了”。标准阿瓦隆允许重复角色，但 agent 分配本质上是玩家维度，而不是角色维度。
- 一旦系统支持重复角色，`ROLE_BINDING` 就天然会把同角色玩家压缩到同一个配置槽位；这对“按阵营/身份选模型”是合理的，对“每个玩家独立 agent”则是错误的。
- 控制台默认入口如果继续给出 `role` 预设，即使底层新增了多人规则，也会把用户持续带回错误的绑定语义。

## ce-plan

- 需要把“按玩家绑定”具体化成“按座位 `seatNo` 绑定”，因为建局时角色尚未分配，但座位已经稳定存在。
- 兼容性策略不能粗暴删除 `ROLE_BINDING`；更稳妥的是新增 `SEAT_BINDING` 作为默认显式选模模式，同时保留 `ROLE_BINDING` 给兼容调用和特殊实验用法。
- 运行时 resolved config 仍应继续按 `playerId` 持久化与恢复，避免把新的外部绑定方式扩散到运行时内部主键。

## ce-review

- 这次改动没有动数据库 schema；新增的 `seatBindings` 只存在于请求 DTO、运行时配置和快照序列化层，所以恢复链路的关键是 `GameSetup` 与 `LlmSelectionConfig` 的序列化兼容。
- `SEAT_BINDING` 的校验边界必须是“只能引用本局 LLM 座位”，否则很容易把模型误绑到 `SCRIPTED` 座位，产生静默错误。
- 控制台展示层也要同步切换到“几号位 -> modelId”，否则即使底层已经修好，操作者仍会从日志里看到旧的角色绑定视角，误以为系统没改对。

## ce-compound

- 当一个系统既有“角色”又有“玩家”两个维度时，凡是会产生独立 agent、独立记忆或独立控制器实例的配置，都应该绑定到玩家维度，而不是角色维度。
- “重复角色”会把很多看似自然的键设计打穿；只要玩法允许同一 `roleId` 出现多次，就不能再把 `roleId` 当成唯一配置键。
- 默认入口的语义和底层能力同样重要。即使 API 已支持正确能力，只要控制台默认项还停留在旧模式，用户仍会稳定走进错误路径。
- 兼容旧模式时，最稳妥的做法通常不是保留旧模式为默认值，而是把它显式降级为兼容选项，并让文档与测试都迁移到新默认值。
