# 04 - 配置与资源文件设计

## 1. 目标

本文件定义项目中需要落地的配置文件与资源文件。第一版不做规则录入 UI，也不做角色管理后台，因此配置文件是规则真相源的载体。

---

## 2. 配置类型总览

第一版至少需要三类配置：

1. 应用配置
2. 规则配置
3. 角色 / 开局模板配置

---

## 3. 应用配置 `application.yml`

建议放在 `avalon-app/src/main/resources/application.yml`。

至少包含：

- Spring Boot 端口
- 数据源
- 默认规则集 ID
- LLM provider api key
- 日志级别
- 审计开关

建议结构：

```yaml
avalon:
  default-rule-set: avalon-classic-5p-v1
  snapshot:
    every-n-events: 10
  audit:
    save-raw-response: true

spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/avalon
    username: postgres
    password: postgres
```

如需接 Spring AI，可继续加入 provider 配置。

---

## 4. 规则配置目录

建议在 `avalon-app/src/main/resources/rules/` 下放置规则文件。

例如：

```text
rules/
├── avalon-classic-5p-v1.yml
├── avalon-classic-6p-v1.yml
└── avalon-classic-7p-v1.yml
```

### 每个规则文件至少描述

- 规则集标识
- 名称与版本
- 最小 / 最大人数
- 轮次任务人数
- 某轮是否需要双失败
- 是否有刺杀阶段
- 刺杀规则
- 可用 setup template 列表

---

## 5. 角色配置目录

建议在 `avalon-app/src/main/resources/roles/` 下放置角色文件，或合并到单一角色目录文件中。

例如：

```text
roles/
├── merlin.yml
├── percival.yml
├── loyal-servant.yml
├── morgana.yml
├── assassin.yml
```

### 每个角色文件至少描述

- roleId
- displayName
- camp
- description
- 能力列表
- 知识规则
- 是否可刺杀
- 是否可加入任务

第一版不必做得过度复杂，但结构上必须支持未来新增角色。

---

## 6. Setup Template 配置目录

建议在 `avalon-app/src/main/resources/setups/` 下放置开局模板。

例如：

```text
setups/
├── classic-5p.yml
├── classic-6p.yml
└── classic-7p.yml
```

### 每个模板至少描述

- templateId
- playerCount
- roleIds

这样规则集可以引用模板，模板再引用角色。

---

## 7. Prompt 模板资源

如果要将 prompt 资源化，建议放在 `avalon-agent/src/main/resources/prompts/`。

例如：

```text
prompts/
├── system/
│   ├── general-system.txt
│   └── response-schema.txt
├── roles/
│   ├── merlin.txt
│   ├── assassin.txt
│   └── loyal-servant.txt
└── phase/
    ├── discussion.txt
    ├── team-proposal.txt
    └── team-vote.txt
```

注意：

- prompt 中不能硬编码规则真相
- 规则摘要应由系统根据 `RuleSetDefinition` 生成后再注入

---

## 8. SQL / Migration 目录

若使用 Flyway，建议：

```text
avalon-app/src/main/resources/db/migration/
├── V1__init_tables.sql
├── V2__add_memory_snapshot.sql
```

第一版建议至少建：

- game_session
- game_player
- game_role_assignment
- game_event
- game_snapshot
- player_memory

---

## 9. 测试资源建议

建议在各模块中保留：

- 示例规则文件
- 示例角色文件
- 示例 setup template
- 测试局数据

这样 Codex / Claude Code 在写单元测试时可以直接引用。

---

## 10. 给 AI 编码代理的要求

1. 所有规则、角色、模板优先 YAML 化
2. 先实现资源加载器，再实现上层服务
3. 资源文件的字段名必须与 Java 模型结构对应
4. 若暂时无法完整支持复杂角色能力，也要预留字段位置
