# 02 - 总体架构设计

## 1. 高层架构

系统分为六大层：

1. 规则定义层
2. 领域模型 + 规则引擎层
3. 运行时编排层
4. Agent 运行层
5. 持久化 / 审计 / 回放层
6. API / 会话 / 交付层

---

## 2. 各层职责

### 2.1 规则定义层
负责维护可版本化的元数据：
- ruleset
- role
- setup template
- 可见性策略
- 任务规则
- 投票规则
- 刺杀规则

### 2.2 领域模型 + 规则引擎层
负责：
- 游戏聚合根
- 阶段状态机
- 动作合法性校验
- 胜负判定
- 可见性计算
- 角色分配服务

要求：
- 尽量保持轻框架
- 不依赖 Spring AI 或 LangChain4j
- 可以单元测试驱动开发

### 2.3 运行时编排层
负责：
- 游戏生命周期
- 当前轮到谁行动
- 解析该调用哪个控制器
- 组装玩家回合上下文
- 调用控制器获得动作
- 持久化结果事件
- 推进到下一个等待态或结束态

### 2.4 Agent 运行层
负责：
- 每个玩家独立模型路由
- 基于结构化上下文构造 Prompt
- 解析结构化响应
- 校验与重试
- 解析记忆更新建议
- 提取审计摘要

### 2.5 持久化 / 审计 / 回放层
负责：
- 事件流
- 局面快照
- 玩家记忆快照
- 单回合审计记录
- 回放投影
- 恢复服务

### 2.6 API / 会话 / 交付层
负责：
- REST 接口
- 未来 SSE / WebSocket 推送
- 运维 / 管理员查看接口
- 真人玩家动作提交接口

## 4. 关键设计原则

### 4.1 规则与智能分离
- 规则引擎只负责真相判断
- LLM 只负责“作为玩家做选择”

### 4.2 状态持久化优先
系统内重要状态不能只存在于内存或 Prompt 中，必须可以落库恢复。

### 4.3 事件驱动优先
未来 UI、回放、评测都依赖事件流，不要只做“当前状态表”。

### 4.4 控制器多态
一切玩家行为都通过统一接口进入系统：
- LLM
- HUMAN
- SCRIPTED

### 4.5 按玩家隔离模型配置
每个席位都允许配置不同 provider 与 model，不允许假定全局只有一个模型。

## 5. 推荐运行流程总览

```text
1. 创建游戏
2. 选择规则集 + 开局模板
3. 创建玩家席位与控制器配置
4. 根据 seed 分配角色
5. 生成每个玩家的私有知识
6. 进入阶段机（讨论 / 组队 / 投票 / 任务 / 刺杀）
7. 每一步都生成事件并更新快照
8. 结束后保留完整回放与审计记录
```

## 6. 给开发 Agent 的任务导向说明

### 你要实现的不是：
- 一个 prompt 工程脚本
- 一个聊天机器人 demo
- 一个强耦合的 5 人局写死程序

### 你要实现的是：
- 一个后端领域清晰、状态可恢复、未来可持续扩展的游戏平台核心

### 架构落地顺序
1. Maven 多模块骨架
2. `avalon-core`
3. `avalon-config`
4. `avalon-runtime`
5. `avalon-persistence`
6. `avalon-api`
7. `avalon-agent`
